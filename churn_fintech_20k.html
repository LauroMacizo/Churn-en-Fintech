Limpieza de datos
Detectar y manejar valores faltantes
El CSV tiene NaN simulados en estimated_salary (~5%), credit_score (~4%) y tenure_months (~3%). Para numéricos imputa con la mediana. Documenta cada decisión en el notebook — no borres filas.
Revisar tipos y rangos
Verifica que age esté entre 18–75, credit_score entre 300–850, complaints entre 0–10. Registra cualquier valor fuera de rango antes de limpiar.
Codificar variables categóricas
gender → Label Encoding (M=0, F=1). Hazlo reproducible con un diccionario explícito, no con LabelEncoder automático, para que el equipo entienda el mapeo.
Feature engineering (exactamente como pide el README)
churn_rate por segmento
Agrupa por products y por is_active. Calcula la tasa de churn de cada grupo con groupby + transform('mean') sobre la columna churn. Agrega la columna churn_rate_segment al dataset limpio.
ratio balance/ingresos
Calcula balance / estimated_salary. Llámala ratio_balance_ingreso. Clientes con ratio alto tienen más "piel en el juego" — hipótesis: churnean menos. Esto lo confirmará Alfonso en el EDA.
engagement score
Combina transacciones y actividad: engagement_score = transactions_last_month * is_active. Clientes activos con muchas transacciones = alto engagement. Clientes inactivos = score 0 independientemente de las transacciones.

¿Los usuarios con menos actividad churnean más?
Usa engagement_score (feature de NAT). Divide en cuartiles y calcula la tasa de churn por cuartil. Grafica barras. La respuesta esperada: sí, el cuartil más bajo debería tener 3–4× más churn que el más alto.
¿Las quejas afectan?
Cruza complaints vs churn. Cada queja adicional suma ~25% de probabilidad de churn (está en el Dataset.py). Muestra esto con un lineplot o barplot. El insight debe decir cuánto, no solo "sí afectan".
¿El balance importa?
Usa ratio_balance_ingreso de NAT. Boxplot separando churn=0 vs churn=1. También cruza si balance > 30,000 tiene menor churn — eso está baked in el dataset y debe ser visible.

Modelos sugeridos (en orden, como indica el README)
Logistic Regression — línea base obligatoria
Empieza aquí. Usa class_weight='balanced' porque el churn no estará al 50/50. Si no supera 0.70 de ROC-AUC en el test set, hay algo mal en los features — revisa con NAT antes de pasar al siguiente modelo.
Random Forest — modelo principal
Activa feature_importances_ de inmediato. Las top-5 variables te dicen qué le importa realmente al modelo — eso es lo que Regi necesita para el pitch. Usa n_estimators=100 como punto de partida.
XGBoost — nivel pro (extra para ganar)
Usa scale_pos_weight = n_no_churn / n_churn para el desbalance. Combina con SHAP para explicar predicciones individuales: "Este cliente tiene 87% de probabilidad de churn porque tiene 4 quejas y 3 transacciones en el último mes."
Métricas (exactamente las del README)
Accuracy / Precision / Recall / ROC-AUC
Reporta las 4 para cada modelo en una tabla comparativa. Prioriza Recall sobre Accuracy — un modelo que no detecta churners es inútil aunque tenga 80% de accuracy. El objetivo mínimo: Recall ≥ 0.70 en la clase churn=1.
En este dataset el churn no estará al 50% — habrá más no-churn que churn. Nunca reportes solo Accuracy. Un modelo que predice "nadie churna" puede tener accuracy alta y ser completamente inútil.
Extras para ganar (del README)

Grafica las top-10 variables con sus importancias. Traduce cada una a lenguaje de negocio para Regi: no "complaints: 0.31" sino "las quejas son el predictor #1 de abandono, por encima del balance o la antigüedad".

Estructura del pitch (exactamente como indica el README)
1 · El problema
¿Cuánto le cuesta el churn a la fintech en números? Estima: si la base es de 20,000 clientes, el ticket promedio mensual es $X y la tasa de churn es Y%, entonces la fintech pierde $Z al mes. Eso abre el pitch con urgencia.
2 · Hallazgos clave
Máximo 3 insights de Alfonso, cada uno con: número concreto + una gráfica + qué significa para el negocio. No leer la gráfica — explicar la consecuencia. Ejemplo del README: "Clientes con menos de 10 transacciones mensuales tienen 4× más probabilidad de churn."
3 · Impacto en negocio
Conecta los hallazgos con dinero o retención. Si reducimos el churn en clientes con 3+ quejas un 20%, ¿cuántos clientes salvamos? Usa los números del dataset para hacer la estimación concreta.
4 · Recomendaciones accionables
Al menos 3 acciones que la fintech podría tomar la semana siguiente. Ejemplo: "Activar una campaña de reenganche para clientes con engagement_score < 5 y sin quejas — son recuperables." Que cada recomendación tenga un responsable implícito (equipo de producto, CX, marketing).
Lo que NO hacer (directo del README)
No leer gráficas sin contexto — cada visual necesita una conclusión en voz alta.
No usar "ROC-AUC", "precision-recall" o "feature importance" sin traducirlos.
No terminar sin una recomendación accionable — los datos sin acción no sirven.
